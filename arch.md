# AGENTS.md

> 本文件定义本仓库中 AI coding agent（如 Codex）的工作规范。
> 除非用户明确要求，否则所有实现、修改、重构、测试、文档更新都必须遵守本文件。

## 1. 项目目标

本项目的目标是：

- [一句话描述产品目标]
- 优先保证：
  1. 可运行
  2. 可维护
  3. 易扩展
- 不追求：
  - 过度设计
  - 提前抽象
  - 无必要的大规模重构

## 2. 工作原则

Agent 在执行任何任务时，必须遵循以下原则：

1. 先理解现有实现，再动手修改。
2. 优先做最小可行改动（minimal diff）。
3. 优先复用现有模块、工具函数、约定，而不是新造轮子。
4. 不得顺手修改与当前任务无关的代码。
5. 不得在没有明确收益的情况下引入新的依赖。
6. 保持变更可测试、可回滚、可解释。
7. 输出结果时说明：
   - 改了什么
   - 为什么这么改
   - 风险点是什么
   - 如何验证

## 3. 代码库理解要求

开始修改前，必须先检查：

- 相关业务模块
- 相关数据结构 / 类型定义
- 相关测试
- 相关配置文件
- 相关文档

如发现现有实现与用户需求冲突，应：

1. 先遵循用户需求
2. 同时指出冲突点
3. 尽量保持向后兼容

## 4. 目录与模块约定

请遵守以下目录规范：

- `src/`：核心业务代码
- `components/`：UI 组件
- `pages/` / `app/`：路由页面
- `services/`：API 调用与服务层
- `hooks/`：复用逻辑
- `utils/`：纯工具函数
- `types/`：公共类型定义
- `tests/`：测试

约束：

- 业务逻辑不要堆在页面层
- API 请求不要直接散落在 UI 组件里
- 工具函数保持纯函数，避免隐藏副作用
- 公共类型集中管理，避免重复定义

## 5. 编码规范

### 通用要求

- 优先写清晰代码，不写炫技代码
- 函数保持短小、单一职责
- 命名必须语义明确，避免 `data`, `temp`, `handleIt` 之类模糊命名
- 尽量消除 magic number，必要时提取常量
- 错误处理必须明确，不能静默吞错
- 日志信息必须可定位问题，避免无意义输出

### 注释要求

- 不要写解释“代码表面行为”的废话注释
- 只在这些场景写注释：
  - 业务规则不直观
  - 为什么这么做不 obvious
  - 临时兼容方案或技术债说明

### TypeScript / JavaScript 要求

- 优先使用 TypeScript
- 尽量避免 `any`
- 公共接口必须有明确类型
- 异步逻辑统一使用 `async/await`
- 不要混用多种风格

## 6. UI / 前端规范

如任务涉及前端，请遵守：

- 优先保证信息层级清晰
- 保持组件职责单一
- 优先复用已有组件与样式 token
- 不随意新增一套视觉风格
- 表单必须有：
  - 校验
  - 错误提示
  - loading 状态
  - 空状态 / 异常状态

除非用户明确要求，否则不要：

- 擅自重做整页 UI
- 引入复杂动画
- 修改全局主题

## 7. 后端 / API 规范

如任务涉及后端，请遵守：

- 明确输入、输出、错误码
- 先做参数校验，再进入业务逻辑
- 不信任客户端输入
- 不在 controller 中堆业务逻辑
- 数据访问逻辑集中到 service / repository 层
- 变更接口时评估兼容性

## 8. 数据与数据库规范

- 不直接修改生产数据逻辑假设
- 涉及 schema 变更时，必须：
  - 提供迁移方案
  - 说明兼容性影响
  - 避免破坏已有数据
- 查询优先保证正确性，再考虑优化
- 不做未经验证的性能“想当然”优化

## 9. 测试与验证要求

完成修改后，必须尽可能验证：

### 至少执行

- lint
- typecheck
- 与本次改动相关的测试

### 如果新增功能

必须补充以下至少一种：

- 单元测试
- 集成测试
- 端到端测试

### 如果没有测试

至少提供：

- 手动验证步骤
- 预期结果
- 风险说明

# AGENTS.md

## Goal
本项目以“快速迭代 + 保持可维护”为目标。
优先交付可运行闭环，避免过度设计。

## Working Rules
- 先阅读相关代码，再开始修改。
- 优先最小可行改动，不做无关重构。
- 复用现有模式、组件、工具函数。
- 不随意新增依赖。
- 不修改与当前任务无关的文件。
- 保持实现简单、可读、可测试。

## Code Style
- 命名清晰，避免模糊命名。
- 函数单一职责，避免超长函数。
- 错误处理明确，不静默吞错。
- 注释解释“为什么”，不要解释显而易见的“做了什么”。
- 优先与现有代码风格保持一致。

## Frontend
- 业务逻辑不要堆在页面层。
- 组件保持单一职责。
- 表单必须有校验、错误提示、loading 状态。

## Backend
- 先校验输入，再进入业务逻辑。
- 不信任客户端输入。
- controller 保持薄，业务逻辑放 service。

## Testing
完成后尽量执行：
- lint
- typecheck
- 与改动相关的测试

如果无法运行测试，必须提供：
- 手动验证步骤
- 预期结果
- 风险说明

## Dependencies
除非必要，不要新增依赖。
新增依赖前先判断：
1. 现有依赖能否完成
2. 原生能力能否完成
3. 是否值得长期维护

## Output Format
每次任务结束时，输出：
- Summary
- Files Changed
- Validation
- Assumptions
- Risks

## For complex tasks
复杂任务先写计划，再实施。
计划可写入 `PLANS.md`。

## Overrides
如果子目录存在更近一级 `AGENTS.md`，则局部规则覆盖全局规则。
